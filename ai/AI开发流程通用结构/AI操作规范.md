# AI 操作规范

本文档定义 AI 操作项目的所有规范，所有 AI 必须严格遵守。

---

## 一、核心原则

> ⚠️ **重要原则：禁止重复定义**
>
> **同一个概念、规则、流程只定义一次**，禁止在多个文档或同一文档的多处重复描述相同内容。
>
> - 详细内容只在专属文档中定义，其他地方通过引用链接
> - AI 在编写文档时，必须检查是否已有相关定义，避免重复
> - 发现重复定义时，应删除冗余内容，保留一处权威定义

### 1.1 修改前必须询问

**禁止直接修改任何代码**。当用户要求修改时，AI 必须：

1. **先分析** - 理解需求和影响范围
2. **先询问** - 确认方案是否符合预期
3. **再执行** - 用户确认后才修改

### 1.2 变更必须同步

任何代码变更，**必须同步更新所有受影响的文件**。详见 `docs/技术规范/代码同步修改规范.md`。

### 1.3 每次操作后必须 commit（强制）

**禁止一次性做多个操作后再 commit**，也**禁止只 commit 代码不 commit 文档**。

每个独立任务完成后**必须立即 commit**：
- 完成一个接口 → 立即 commit
- 完成一个文档 → 立即 commit
- 完成一个配置修改 → 立即 commit

详见 `docs/技术规范/Git规范.md`。

### 1.4 接口必须先写文档（强制）

**禁止先写代码后补文档**，也**禁止不写接口文档**。

每个接口开发时，必须：
1. **先编写** `@swag_from` 装饰器（包含 summary、description、parameters、responses）
2. **再编写**视图函数实现
3. **完成后**运行导出脚本生成可读文档

详见 `docs/技术规范/Flask后端规范.md` 第13章。

## 三、技术选型确定

### 3.1 技术选型询问

当进入**设计阶段**时，AI 必须询问用户确定技术选型：

```markdown
## 技术选型确认

请确认项目使用的技术栈：

### 后端技术（可选）
- [ ] Flask (Python)
- [ ] Node.js (Express/Koa/NestJS)
- [ ] Go (Gin/Fiber)
- [ ] Java (Spring Boot)
- [ ] 其他：

### 前端技术（可选）
- [ ] Vue 3
- [ ] React
- [ ] 小程序
- [ ] 其他：

### 其他技术
- [ ] 数据库：MySQL / PostgreSQL / MongoDB
- [ ] 缓存：Redis / Memcached
- [ ] 消息队列：（根据需要）

请确认后将在设计文档中明确技术选型。
```

### 3.2 技术规范识别

确定技术选型后，AI 自动识别对应的技术规范：

1. **检查 `docs/技术规范/` 目录**
   - 若有对应的技术规范 → 询问用户"是否遵守该技术规范？"
     - 若**遵守** → 在设计文档中注明，后续严格遵循
     - 若**不遵守** → 说明需要遵守的规范内容，由用户确认
   - 若**没有**对应的技术规范 → 询问用户"是否需要自动创建该技术规范？"
     - 若**需要** → 基于通用最佳实践创建技术规范文档

2. **技术规范范围**
   - 后端技术（Flask/Node.js/Go/Java等）→ 对应后端规范
   - 前端技术（Vue/React/小程序等）→ 对应前端规范
   - 数据库/缓存/消息队列 → 可选，需根据项目实际情况

## 五、强制阅读清单

开始任何任务前，**必须先完整阅读**：

1. `AI操作规范.md` - 所有规范的总览
2. `docs/技术规范/Git规范.md` - Git使用规范（**最早识别，后续commit用**）
3. `docs/原始需求/` - 原始需求（如有）
4. `docs/需求文档/` - 需求文档（如有）
5. `docs/设计文档/` - 项目技术选型（进入设计阶段前，**需用户确认**）
6. `docs/技术规范/` - 遵循的技术标准（进入代码阶段前）
7. `docs/细节记录/` - 项目重要细节（如有）

## 六、AI 执行 Checklist

收到任务时，按顺序执行：

1. **识别项目核心规范**（**强制遵守，无需询问**）
   - Git 规范：`docs/技术规范/Git规范.md`（**后续每次操作后按规范强制 commit**）
   - 确定项目类型：后端/前端/全栈
   - 识别对应的技术规范（如 `docs/技术规范/Flask后端规范.md`）

2. **查找需求**
   - 检查 `docs/原始需求/` 和 `docs/需求文档/`

3. **确认设计文档状态**
   - 若有设计文档，**必须询问用户**："设计文档中的技术选型是否已经您明确确认？"
     - 若**未确认** → 询问用户确认技术选型 → 更新设计文档 → 继续
     - 若**已确认** → 继续执行
   - ⚠️ **注意**：设计文档存在 ≠ 用户已确认（可能是模板或占位内容）

4. **按流程执行**
   - 有原始需求，无需求文档 → 转换需求文档 → 保存到 `docs/需求文档/`
   - 有需求文档，无设计文档 → 询问用户技术选型 → 生成设计文档 → 保存到 `docs/设计文档/`
   - 有设计文档（且已确认）→ 确定遵循的技术规范 → 生成代码

5. **记录细节**：重要细节同步记录到 `docs/细节记录/`

6. **保持更新**：文档随项目进展持续更新

## 七、路径变更原则

**当任何路径变更时，必须同步更新所有引用该路径的文件**。

> **注意**：文档中的路径示例使用相对路径或占位符（如 `<项目路径>/apps`），实际项目根据具体情况调整。AGENTS.md 中的绝对路径是 AI Coding 工具配置所需，无需改动。

| 路径类型 | 示例 | 必须更新的位置 |
|:---------|:-----|:--------------|
| 工作目录 | `<项目路径>/backend` | AGENTS.md |
| 文档目录 | `docs/` | AGENTS.md、文档内互相引用 |
| 代码目录 | `apps/` | `__init__.py` 导入 |

**禁止**：
- ❌ 只改目录名，不更新引用
- ❌ 只改文档，不改代码
- ❌ 只改代码引用，不改文档

## 八、临时文件管理

AI 生成的临时文件**必须**放在 `temp/` 目录，用完即删。

| 类型 | 示例 | 处理 |
|:-----|:-----|:-----|
| 调试输出 | `temp/debug_*.json` | 用完即删 |
| 测试数据 | `temp/test_*.sql` | 用完即删 |
| 临时脚本 | `temp/analysis_*.py` | 用完即删 |
| 导出文件 | `temp/export_*.xlsx` | 用户下载后删 |

**禁止**：
- ❌ 在项目根目录或代码目录下创建临时文件
- ❌ 临时文件提交到 git
- ✅ 用完立即删除，不堆积

## 十一、文档编写原则

### 11.1 不冗余原则

**禁止在多个文档中重复定义相同内容**。每个知识点只在一个地方定义，其他地方通过引用链接。

| 错误做法 | 正确做法 |
|:---------|:---------|
| 在多个文档中写相同的概念定义 | 概念只定义一次，其他地方引用 |
| 在 AI操作规范 中写详细的技术细节 | 详细内容写在技术规范中，AI操作规范只引用 |
| 在多个地方复制粘贴相同的说明 | 写一份，其他地方引用 |

### 11.2 不重复定义原则

**同一个概念、规则、流程只定义一次**，无论是在单个文档内部还是多个文档之间。

**单个文档内部**：
- 同一概念只在文档的一处定义
- 相似内容合并为一处，避免分散描述
- 避免"在第3章讲过，第5章又讲一遍"

**多个文档之间**：
- 详细内容只在专属文档中定义，其他文档只引用
- 避免在多个文档中复制相同的说明

### 11.3 内容完整原则

**每个文档的内容必须是完整的**，避免：

- 文档内容残缺，需要参照其他文档才能理解
- 关键信息缺失，需要猜测或询问
- 示例不完整，无法直接使用

### 11.4 编写检查清单

编写新文档或更新文档时，对照检查：

- [ ] 是否有内容在其他文档中重复定义？
- [ ] 是否有内容应该引用而非复制？
- [ ] 文档内容是否完整，可以独立使用？
- [ ] 示例是否完整可运行？
- [ ] 引用路径是否正确？
