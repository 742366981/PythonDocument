# AI 操作规范

本文档定义 AI 操作项目的所有规范，所有 AI 必须严格遵守。

---

## 零、速查表（每次任务前必须对照）

> ⚠️ **AI 每次任务开始前，必须先确认以下事项**

### 强制执行项
- [ ] **修改前**：必须先分析影响、询问确认
- [ ] **完成时**：必须立即 commit（禁止多个操作后才 commit）
- [ ] **接口开发**：必须先写 docstring，再写实现
- [ ] **代码注释**：函数/类/复杂逻辑必须有完整注释
- [ ] **临时文件**：必须放在 `temp/` 目录，用完即删

### 禁止项
- ❌ 未经用户确认直接修改任何代码/文档
- ❌ 先写代码后补文档
- ❌ 只 commit 代码不 commit 文档
- ❌ 多个文档重复定义相同内容
- ❌ 在项目目录创建临时文件

## 一、核心原则（强制约束）

> ⚠️ **强制约束（红线）**
>
> 以下行为是 AI 操作的**红线**，违反将被视为不合格：

| 红线行为 | 说明 | 违规后果 |
|:---------|:-----|:---------|
| **未经确认直接修改代码/文档** | 用户未确认前不得执行任何修改 | 用户有权要求回滚 |
| **先写代码后补文档** | 接口必须先写 docstring 再写实现 | 视为不合格 |
| **多个操作后才 commit** | 每个独立任务完成后必须立即 commit | 视为不规范 |
| **只 commit 代码不 commit 文档** | 代码和文档变更必须同时提交 | 视为不规范 |
| **发现重复定义未处理** | 冗余内容必须删除，保留一处 | 视为不合格 |
| **代码注释不完整** | 所有代码必须有完整注释（见本章第1.5节） | 视为不合格 |
| **临时文件不清理** | 调试文件用完即删，不堆积 | 视为不规范 |

### 1.1 修改前必须询问

**禁止直接修改任何代码/文档**。当用户要求修改时，AI 必须：

1. **先分析** - 理解需求和影响范围
2. **先询问** - 确认方案是否符合预期
3. **再执行** - 用户确认后才修改

### 1.2 变更必须同步

任何代码变更，**必须同步更新所有受影响的文件**。详见 `docs/技术规范/代码同步修改规范.md`。

### 1.3 每次操作后必须 commit（强制）

**禁止一次性做多个操作后再 commit**，也**禁止只 commit 代码不 commit 文档**。

每个独立任务完成后**必须立即 commit**：
- 完成一个接口 → 立即 commit
- 完成一个文档 → 立即 commit
- 完成一个配置修改 → 立即 commit

详见 `docs/技术规范/Git规范.md`。

### 1.4 接口必须先写文档（强制）

**禁止先写代码后补文档**，也**禁止不写接口文档**。

每个接口开发时，必须：
1. **检查模板文件** - 详见 `docs/技术规范/Flask后端规范.md` 第13章
2. **编写**视图函数docstring - 详见 `docs/技术规范/Flask后端规范.md` 第13章
3. **编写**视图函数实现
4. **完成后**运行导出脚本生成可读文档

### 1.5 代码注释必须完整（强制）

**所有代码必须有完整注释**，包括但不限于：

| 代码类型 | 注释要求 | 示例 |
|:---------|:---------|:-----|
| **函数/方法** | 必须有 docstring，说明功能、参数、返回值 | `"""获取用户信息\n\nArgs:\n    user_id: 用户ID\n\nReturns:\n    User: 用户对象\n"""` |
| **类定义** | 必须有类注释，说明用途 | `"""用户模型类\n\n用于管理用户数据的增删改查操作\n"""` |
| **复杂逻辑** | 关键判断、分支必须有注释 | `# 超过阈值才入库，避免噪音数据` |
| **常量定义** | 必须有注释说明用途 | `# 每页返回的最大记录数` |
| **配置文件** | 必须有注释说明配置项含义 | `# 数据库连接池大小` |

**注释语言**：必须与现有代码库保持一致（自动检测）。

**禁止**：
- ❌ 空函数无注释
- ❌ 复杂逻辑无注释说明
- ❌ 配置项无注释
- ❌ 用注释替代完整文档

## 二、边界场景处理

| 场景 | 处理方式 |
|:-----|:---------|
| 用户说"快点改一下" | 仍需询问确认，但可以简化确认方式（如"确认修改X？Y/N"） |
| 紧急修复bug | 仍需先询问，可以加快确认速度 |
| 规范之间冲突 | 优先顺序：核心约束（红线）> 禁止行为 > 建议做法 |
| 不确定是否需要修改 | 先询问用户，不要自行判断 |

## 四、技术选型确定

### 4.1 技术选型询问

当进入**设计阶段**时，AI 必须询问用户确定技术选型：

```markdown
## 技术选型确认

请确认项目使用的技术栈：

### 后端技术（可选）
- [ ] Flask (Python)
- [ ] Node.js (Express/Koa/NestJS)
- [ ] Go (Gin/Fiber)
- [ ] Java (Spring Boot)
- [ ] 其他：

### 前端技术（可选）
- [ ] Vue 3
- [ ] React
- [ ] 小程序
- [ ] 其他：

### 其他技术
- [ ] 数据库：MySQL / PostgreSQL / MongoDB
- [ ] 缓存：Redis / Memcached
- [ ] 消息队列：（根据需要）

请确认后将在设计文档中明确技术选型。
```

### 4.2 技术规范识别

确定技术选型后，AI 自动识别对应的技术规范：

1. **检查 `docs/技术规范/` 目录**
   - 若有对应的技术规范 → 询问用户"是否遵守该技术规范？"
     - 若**遵守** → 在设计文档中注明，后续严格遵循
     - 若**不遵守** → 说明需要遵守的规范内容，由用户确认
   - 若**没有**对应的技术规范 → 询问用户"是否需要自动创建该技术规范？"
     - 若**需要** → 基于通用最佳实践创建技术规范文档

2. **技术规范范围**
   - 后端技术（Flask/Node.js/Go/Java等）→ 对应后端规范
   - 前端技术（Vue/React/小程序等）→ 对应前端规范
   - 数据库/缓存/消息队列 → 可选，需根据项目实际情况

### 7.2 步骤详细说明

#### Step 0：新项目初始化（仅首次/缺失文件时执行）

**触发条件**：项目根目录**缺失**以下任一文件时
- `.gitignore`
- 语言特定文件（Python: requirements.txt, Node.js: package.json 等）
- 虚拟环境脚本（venv_windows.bat 等）

**跳过条件**：以上文件全部存在 → 直接进入 Step 1

**执行流程**：
1. **先询问** - 向用户确认是否需要初始化：
   ```
   ## 新项目初始化确认

   检测到项目缺少以下文件：
   - [缺失文件列表]

   是否需要按开发环境规范执行初始化？
   - [ ] 是，执行初始化（创建 .gitignore、语言特定文件、虚拟环境脚本等）
   - [ ] 否，跳过初始化

   请确认。
   ```
2. **用户确认后** - 按 `docs/技术规范/开发环境规范.md` 执行：
   1. 创建 .gitignore（根据项目类型识别）
   2. 创建语言特定文件
   3. 创建虚拟环境脚本
   4. 创建环境配置示例
   5. 更新记忆文件

**自检确认**：
- [ ] `.gitignore` 是否已创建？ → 验证：`ls .gitignore`
- [ ] 语言特定文件是否已创建？ → 验证：`ls requirements.txt`（或对应语言文件）
- [ ] 虚拟环境脚本是否已创建？ → 验证：`ls venv_windows.bat`
- [ ] 记忆文件是否已更新？ → 验证：`cat .claude/memory/*.md`

---

#### Step 1：识别核心规范（**强制执行**）

**触发条件**：**始终执行**

**执行内容**：
- Git 规范：`docs/技术规范/Git规范.md`（**后续每次操作后按规范强制 commit**）
- 确定项目类型：后端 / 前端 / 全栈
- 识别对应的技术规范（如 `docs/技术规范/Flask后端规范.md`）

**自检确认**：
- [ ] 是否已完整阅读Git规范？ → 验证：能复述commit规范要点
- [ ] 项目类型是否已识别（后端/前端/全栈）？ → 验证：明确项目类型
- [ ] 对应的技术规范文件是否已定位？ → 验证：`ls docs/技术规范/*.md`
- [ ] commit的规范要求是否清楚？ → 验证：能说明commit时机和格式

⚠️ 若有任何一项为"否"，必须先完成该项才能继续。

---

#### Step 2：查找需求

**触发条件**：**始终执行**

**执行内容**：
- 检查 `docs/原始需求/` 是否有原始需求
- 检查 `docs/需求文档/` 是否有已转换的需求文档

**自检确认**：
- [ ] `docs/原始需求/` 目录是否已检查？ → 验证：`ls docs/原始需求/`
- [ ] `docs/需求文档/` 目录是否已检查？ → 验证：`ls docs/需求文档/`
- [ ] 需求状态是否已明确（无/已有但需转换/已完整）？ → 验证：明确当前状态

---

#### Step 3：确认设计文档

**触发条件**：**始终执行**

**执行内容**：
- 若**无设计文档** → 进入需求转换阶段 → 生成设计文档 → 询问用户确认技术选型
- 若**有设计文档但未确认** → 询问用户确认技术选型 → 更新设计文档
- 若**有设计文档且已确认** → 直接进入 Step 4

⚠️ **注意**：设计文档存在 ≠ 用户已确认（可能是模板或占位内容）

**自检确认**：
- [ ] 设计文档是否存在？ → 验证：`ls docs/设计文档/*.md`
- [ ] 用户是否已明确技术选型？ → 验证：用户已确认
- [ ] 若需要确认，是否已向用户询问？ → 验证：已发送询问

---

#### Step 4：按流程执行

根据 Step 2-3 的结果，按第5章"完整项目执行流程"执行：

| 当前状态 | 对应阶段 | 执行内容 |
|:---------|:---------|:---------|
| 有原始需求，无需求文档 | 阶段1：需求转换 | 读取 `docs/原始需求/` → 转换为标准化需求文档 → 保存到 `docs/需求文档/` |
| 有需求文档，无设计文档 | 阶段2：设计生成 | 询问用户确认技术选型 → 生成设计文档 → 保存到 `docs/设计文档/` |
| 有设计文档（已确认） | 阶段3：代码生成 | 确定遵循的技术规范 → 按技术规范生成代码 |
| 代码生成完成后 | 阶段4：细节记录 | 记录重要细节到 `docs/细节记录/` |

**每个阶段自检确认**：
- 阶段1需求转换完成：
  - [ ] 需求文档是否完整？ → 验证：包含项目概述、系统定位、功能需求
  - [ ] 是否已保存到 `docs/需求文档/`？ → 验证：`ls docs/需求文档/*.md`
- 阶段2设计生成完成：
  - [ ] 设计文档是否包含技术选型、系统架构？ → 验证：文档内容完整
  - [ ] 是否已获得用户确认？ → 验证：用户已确认
- 阶段3代码生成完成：
  - [ ] 是否每个接口都先写了docstring？ → 验证：检查视图函数
  - [ ] 是否每个函数/类都有完整注释？ → 验证：检查代码注释
  - [ ] 是否符合技术规范？ → 验证：参数命名、字段注释等
  - [ ] 是否按要求commit？ → 验证：`git log --oneline -1`
- 阶段4细节记录完成：
  - [ ] 本次开发是否涉及用户告知的关键信息？ → 验证：询问用户或检查记录
  - [ ] 如果是，是否已在 `docs/细节记录/` 中记录？ → 验证：`ls docs/细节记录/`

**注意**：每个阶段完成后必须立即 commit（详见第一章第1.3节）

#### Step 5：记录细节

**触发条件**：用户主动告知关键信息时

**执行内容**：
- 用户告知的场景/限制/注意事项 → 立即记录到 `docs/细节记录/`
- 用户告知的技术要点/业务规则 → 立即记录到 `docs/细节记录/`

**记录格式**：
```markdown
## 用户告知的关键信息

| 日期 | 内容 | 用途/关联 |
|-----|------|----------|
| 2026-05-09 | 用户告知的XXX | 用于XXX功能开发 |
```

**禁止**：
- ❌ AI 自己判断"重要"而记录
- ❌ 记录显而易见的常识

**开发前检查**：
- 修改某功能前 → 检查是否有该功能相关的用户告知记录
- 有记录 → 必须遵守记录内容
- 无记录 → 正常开发

## 八、路径变更原则

**当任何路径变更时，必须同步更新所有引用该路径的文件**。

| 路径类型 | 示例 | 必须更新的位置 |
|:---------|:-----|:--------------|
| 工作目录 | `{项目名}` | AGENTS.md |
| 文档目录 | `docs/` | AGENTS.md、文档内互相引用 |
| 代码目录 | `apps/` | `__init__.py` 导入 |

**禁止**：
- ❌ 只改目录名，不更新引用
- ❌ 只改文档，不改代码
- ❌ 只改代码引用，不改文档

## 九、临时文件管理

AI 生成的临时文件**必须**放在 `temp/` 目录，用完即删。

| 类型 | 示例 | 处理 |
|:-----|:-----|:-----|
| 调试输出 | `temp/debug_*.json` | 用完即删 |
| 测试数据 | `temp/test_*.sql` | 用完即删 |
| 临时脚本 | `temp/analysis_*.py` | 用完即删 |
| 导出文件 | `temp/export_*.xlsx` | 用户下载后删 |

**禁止**：
- ❌ 在项目根目录或代码目录下创建临时文件
- ❌ 临时文件提交到 git
- ✅ 用完立即删除，不堆积

## 十、文档编写原则

### 10.1 不冗余原则

**禁止在多个文档中重复定义相同内容**。每个知识点只在一个地方定义，其他地方通过引用链接。

| 错误做法 | 正确做法 |
|:---------|:---------|
| 在多个文档中写相同的概念定义 | 概念只定义一次，其他地方引用 |
| 在 AI操作规范 中写详细的技术细节 | 详细内容写在技术规范中，AI操作规范只引用 |
| 在多个地方复制粘贴相同的说明 | 写一份，其他地方引用 |

### 10.2 不重复定义原则

**同一个概念、规则、流程只定义一次**，无论是在单个文档内部还是多个文档之间。

**单个文档内部**：
- 同一概念只在文档的一处定义
- 相似内容合并为一处，避免分散描述
- 避免"在第3章讲过，第5章又讲一遍"

**多个文档之间**：
- 详细内容只在专属文档中定义，其他文档只引用
- 避免在多个文档中复制相同的说明

### 10.3 内容完整原则

**每个文档的内容必须是完整的**，避免：

- 文档内容残缺，需要参照其他文档才能理解
- 关键信息缺失，需要猜测或询问
- 示例不完整，无法直接使用

### 10.4 编写检查清单

编写新文档或更新文档时，对照检查：

- [ ] 是否有内容在其他文档中重复定义？
- [ ] 是否有内容应该引用而非复制？
- [ ] 文档内容是否完整，可以独立使用？
- [ ] 示例是否完整可运行？
- [ ] 引用路径是否正确？
