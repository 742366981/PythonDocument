> ⚠️ **重要说明**
>
> 本文档是 AI 操作的**总规范**，本文档本身的所有章节都需要加强制标注。
> 各技术规范文档的章节标注应参照本文档格式执行。

## 0. 速查表（强制）

> ⚠️ **AI 每次任务开始前，必须先确认以下事项**

### 强制执行项
- [ ] **修改前**：必须先分析影响、询问确认
- [ ] **修改时**：必须遵守 `docs/技术规范/代码同步修改规范.md`
- [ ] **完成时**：必须立即 commit（禁止多个操作后才 commit）
- [ ] **接口开发**：必须先写 docstring，再写实现
- [ ] **代码注释**：函数/类/复杂逻辑必须有完整注释
- [ ] **临时文件**：必须放在 `temp/` 目录，用完即删

### 禁止项
- ❌ 未经用户确认直接修改任何代码/文档
- ❌ 先写代码后补文档
- ❌ 只 commit 代码不 commit 文档
- ❌ 多个文档重复定义相同内容
- ❌ 在项目目录创建临时文件

## 1. 核心原则（强制）

> ⚠️ **强制（红线）**
>
> 以下行为是 AI 操作的**红线**，违反将被视为不合格：

| 红线行为 | 说明 | 违规后果 |
|:---------|:-----|:---------|
| **未经确认直接修改代码/文档** | 用户未确认前不得执行任何修改 | 用户有权要求回滚 |
| **先写代码后补文档** | 接口必须先写 docstring 再写实现 | 视为不合格 |
| **多个操作后才 commit** | 每个独立任务完成后必须立即 commit | 视为不规范 |
| **只 commit 代码不 commit 文档** | 代码和文档变更必须同时提交 | 视为不规范 |
| **发现重复定义未处理** | 冗余内容必须删除，保留一处 | 视为不合格 |
| **代码注释不完整** | 所有代码必须有完整注释（见本章第1.5节） | 视为不合格 |
| **临时文件不清理** | 调试文件用完即删，不堆积 | 视为不规范 |

### 1.1 修改前必须询问

**禁止直接修改任何代码/文档**。当用户要求修改时，AI 必须：

1. **先分析** - 理解需求和影响范围
2. **先检查** - 查阅 `docs/技术规范/代码同步修改规范.md`，确认需要同步哪些文件
3. **先询问** - 确认方案是否符合预期
4. **再执行** - 用户确认后才修改

### 1.2 变更必须同步

任何代码变更，**必须同步更新所有受影响的文件**。详见 `docs/技术规范/代码同步修改规范.md`。

### 1.3 每次操作后必须 commit（强制）

**禁止一次性做多个操作后再 commit**，也**禁止只 commit 代码不 commit 文档**。

每个独立任务完成后**必须立即 commit**：
- 完成一个接口 → 立即 commit
- 完成一个文档 → 立即 commit
- 完成一个配置修改 → 立即 commit

详见 `docs/技术规范/Git规范.md`。

### 1.4 接口必须先写文档（强制）

**禁止先写代码后补文档**，也**禁止不写接口文档**。

每个接口开发时，必须：
1. **检查模板文件** - 详见 `docs/技术规范/Flask后端规范.md` 第13章
2. **编写**视图函数docstring - 详见 `docs/技术规范/Flask后端规范.md` 第13章
3. **编写**视图函数实现
4. **完成后**运行导出脚本生成可读文档

### 1.5 代码注释必须完整（强制）

**所有代码必须有完整注释**，包括但不限于：

| 代码类型 | 注释要求 | 示例 |
|:---------|:---------|:-----|
| **函数/方法** | 必须有 docstring，说明功能、参数、返回值 | `"""获取用户信息\n\nArgs:\n    user_id: 用户ID\n\nReturns:\n    User: 用户对象\n"""` |
| **类定义** | 必须有类注释，说明用途 | `"""用户模型类\n\n用于管理用户数据的增删改查操作\n"""` |
| **复杂逻辑** | 关键判断、分支必须有注释 | `# 超过阈值才入库，避免噪音数据` |
| **常量定义** | 必须有注释说明用途 | `# 每页返回的最大记录数` |
| **配置文件** | 必须有注释说明配置项含义 | `# 数据库连接池大小` |

**注释语言**：必须与现有代码库保持一致（自动检测）。

**禁止**：
- ❌ 空函数无注释
- ❌ 复杂逻辑无注释说明
- ❌ 配置项无注释
- ❌ 用注释替代完整文档

## 2. 边界场景处理（强制）

| 场景 | 处理方式 |
|:-----|:---------|
| 用户说"快点改一下" | 仍需询问确认，但可以简化确认方式（如"确认修改X？Y/N"） |
| 紧急修复bug | 仍需先询问，可以加快确认速度 |
| 规范之间冲突 | 优先顺序：核心约束（红线）> 禁止行为 > 建议做法 |
| 不确定是否需要修改 | 先询问用户，不要自行判断 |

## 4. 技术选型确定（强制）

### 4.1 技术选型询问

当进入**设计阶段**时，AI 必须询问用户确定技术选型：

```markdown
## 技术选型确认

请确认项目使用的技术栈：

### 后端技术（可选）
- [ ] Flask (Python)
- [ ] Node.js (Express/Koa/NestJS)
- [ ] Go (Gin/Fiber)
- [ ] Java (Spring Boot)
- [ ] 其他：

### 前端技术（可选）
- [ ] Vue 3
- [ ] React
- [ ] 小程序
- [ ] 其他：

### 其他技术
- [ ] 数据库：MySQL / PostgreSQL / MongoDB
- [ ] 缓存：Redis / Memcached
- [ ] 消息队列：（根据需要）

请确认后将在设计文档中明确技术选型。
```

### 4.2 技术规范识别

确定技术选型后，AI 自动识别对应的技术规范：

1. **检查 `docs/技术规范/` 目录**
   - 若有对应的技术规范 → 询问用户"是否遵守该技术规范？"
     - 若**遵守** → 在设计文档中注明，后续严格遵循
     - 若**不遵守** → 说明需要遵守的规范内容，由用户确认
   - 若**没有**对应的技术规范 → 询问用户"是否需要自动创建该技术规范？"
     - 若**需要** → 基于通用最佳实践创建技术规范文档

2. **技术规范范围**
   - 后端技术（Flask/Node.js/Go/Java等）→ 对应后端规范
   - 前端技术（Vue/React/小程序等）→ 对应前端规范
   - 数据库/缓存/消息队列 → 可选，需根据项目实际情况

## 5.1 三阶段同步机制（强制）

需求、设计、代码三者必须保持同步，任何变更都必须向上下游传递。

### 5.1.1 同步关系图

```
┌──────────────────────────────────────────────────────────────┐
│                     三阶段同步机制                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌──────────┐      转换      ┌──────────┐     转化     ┌──────────┐     生成     ┌──────────┐     │
│   │ 原始需求  │ ───────────→ │ 标准需求  │ ──────────→ │ 设计文档  │ ──────────→ │   代码   │     │
│   └──────────┘              └──────────┘              └──────────┘              └──────────┘     │
│                                                              │
│         ↑                                    │                                    │                │
│         │              变更同步              ↓                                    │                │
│         │ ───────────────────────────────── ────────────────────────────────── →                │
│                                                              │
│         ↑                                                         │                │
│         │                    代码逆向修正（当代码实现方式与设计文档 │                │
│         │                    不一致时，以实际代码实现更新设计文档） │                │
│                                                              │
│   任意环节变更，都必须同步更新下游所有环节                        │
└──────────────────────────────────────────────────────────────┘
```

### 5.1.2 同步方向（强制）

| 同步方向 | 触发条件 | 同步内容 |
|:---------|:---------|:---------|
| **正向同步** | 需求变更 | 原始需求 → 标准需求 → 设计文档 → 代码 |
| **逆向修正** | 代码实现与设计文档不符 | 代码实现 → 更新设计文档 |

### 5.1.3 正向同步规则（强制）

| 变更环节 | 必须同步的下游 | 同步内容 |
|:---------|:---------------|:---------|
| 原始需求变更 | 标准需求 | 更新需求文档对应内容，标注变更点 |
| 标准需求变更 | 设计文档 | 更新设计文档对应内容，说明变更原因 |
| 设计文档变更 | 代码 | 更新代码实现，保持与设计文档一致 |

**正向同步必须按顺序执行，禁止跳级**：
```
需求变更 → 不允许直接改代码，必须先更新设计文档
```

### 5.1.4 逆向修正规则（强制）

| 触发情形 | 修正内容 | 说明 |
|:---------|:---------|:-----|
| 代码实现方式与设计文档描述不符 | 更新设计文档 | 以实际代码实现为准，修正设计文档 |
| 代码发现更优方案 | 先更新设计文档，再改代码 | 不得擅自改变设计，必须先走变更流程 |

**逆向修正的判断标准**：
- 代码实现与设计文档功能一致，但实现方式不同 → 更新设计文档以匹配代码
- 代码实现缺失设计文档中的功能 → 必须补全代码实现
- 代码实现超出设计文档范围 → 需评估是否属于需求变更

### 5.1.5 禁止行为（强制）

| 禁止行为 | 说明 | 违规后果 |
|:---------|:-----|:---------|
| ❌ 代码与设计文档不一致 | 代码实现必须符合设计文档 | 视为不合格 |
| ❌ 设计文档与需求不一致 | 设计文档必须实现需求文档的内容 | 视为不合格 |
| ❌ 需求文档与原始需求不一致 | 需求文档必须准确反映原始需求 | 视为不合格 |
| ❌ 变更只改一处 | 变更必须同步所有下游文档和代码 | 视为不规范 |
| ❌ 跳过设计文档直接改代码 | 需求变更必须先更新设计文档 | 视为不合格 |
| ❌ 代码完成后不验证设计文档 | 每次代码变更后必须检查设计文档 | 视为不规范 |

### 5.1.6 变更处理流程（强制）

当任何环节发生变更时，必须按以下流程处理：

```
┌─────────────────────────────────────────────────────────────┐
│                    变更处理流程                              │
├─────────────────────────────────────────────────────────────┤
│  1. 评估影响                                                │
│     - 确定变更涉及的环节（需求/设计/代码）                   │
│     - 确定同步范围（哪些文档和代码需要更新）                 │
├─────────────────────────────────────────────────────────────┤
│  2. 执行同步                                                │
│     - 正向变更：按顺序更新需求→设计→代码                     │
│     - 逆向修正：更新设计文档以匹配代码实现                    │
│     - 同步时在变更处标注"【变更】"标记                       │
├─────────────────────────────────────────────────────────────┤
│  3. 验证一致                                                │
│     - [ ] 代码实现的功能点与设计文档一致                     │
│     - [ ] 设计文档的功能点与需求文档一致                     │
│     - [ ] 需求文档准确反映原始需求                           │
├─────────────────────────────────────────────────────────────┤
│  4. 记录变更                                                │
│     - 在变更处的注释中说明变更原因和同步关系                 │
│     - commit 时在 message 中标注涉及的同步变更              │
└─────────────────────────────────────────────────────────────┘
```

### 5.1.7 同步验证检查清单（强制）

每次变更完成后，必须逐项验证：

**正向同步验证**：
- [ ] 需求变更是否已同步到标准需求文档？
- [ ] 标准需求变更是否已同步到设计文档？
- [ ] 设计文档变更是否已同步到代码实现？

**逆向修正验证**：
- [ ] 代码实现与设计文档描述是否一致？
- [ ] 如有不一致，是否已更新设计文档？

**禁止项验证**：
- [ ] 是否有跳级变更（需求→代码，而没有更新设计）？
- [ ] 是否有只改代码而未验证设计文档的情况？

### 5.1.8 同步示例

**场景1：用户要求修改用户表结构**

```
用户要求：用户表新增"邮箱"字段

1. 评估影响
   - 影响：标准需求文档（字段说明）、设计文档（表结构）、代码（models）

2. 执行同步
   - [变更] 标准需求文档：用户表字段新增 email 字段说明
   - [变更] 设计文档：用户表结构新增 email 字段
   - [变更] 代码实现：User 模型新增 email 字段

3. 验证一致
   - [验证] 代码中 email 字段与设计文档一致
   - [验证] 设计文档的表结构与需求文档一致

4. 记录变更
   - 代码注释：[变更] 用户表新增 email 字段，关联需求变更 #xxx
```

**场景2：代码实现与设计文档不一致**

```
发现：代码中 User 模型有 avatar 字段，但设计文档未记录

1. 评估影响
   - 影响：设计文档（缺少 avatar 字段描述）

2. 执行同步
   - [修正] 设计文档：补充 avatar 字段说明

3. 验证一致
   - [验证] 代码实现与设计文档一致

4. 记录变更
   - 设计文档注释：[逆向修正] 补充代码中已实现的 avatar 字段
```

## 8. 路径变更原则（强制）

**当任何路径变更时，必须同步更新所有引用该路径的文件**。

| 路径类型 | 示例 | 必须更新的位置 |
|:---------|:-----|:--------------|
| 工作目录 | `{项目名}` | CLAUDE.md、AGENTS.md |
| 文档目录 | `docs/` | CLAUDE.md、AGENTS.md、文档内互相引用 |
| 代码目录 | `apps/` | `__init__.py` 导入 |

**禁止**：
- ❌ 只改目录名，不更新引用
- ❌ 只改文档，不改代码
- ❌ 只改代码引用，不改文档

### 8.1 索引同步原则（强制）

**新增、修改或删除规范文档时，必须同步更新所有引用了该规范的文档**：

| 变更类型 | 必须更新的文件 |
|:---------|:---------------|
| 新增规范文档 | 所有引用该规范的文档（CLAUDE.md、AGENTS.md、README.md、AI操作规范.md等） |
| 修改规范文档 | 同上，更新对应引用说明 |
| 删除规范文档 | 同上，移除对应条目 |

**禁止**：
- ❌ 新增文档但不更新任何引用
- ❌ 修改文档但不更新引用说明
- ❌ 删除文档但不移除引用

## 9. 临时文件管理（强制）

AI 生成的临时文件**必须**放在 `temp/` 目录，用完即删。

| 类型 | 示例 | 处理 |
|:-----|:-----|:-----|
| 调试输出 | `temp/debug_*.json` | 用完即删 |
| 测试数据 | `temp/test_*.sql` | 用完即删 |
| 临时脚本 | `temp/analysis_*.py` | 用完即删 |
| 导出文件 | `temp/export_*.xlsx` | 用户下载后删 |

**禁止**：
- ❌ 在项目根目录或代码目录下创建临时文件
- ❌ 临时文件提交到 git
- ✅ 用完立即删除，不堆积

## 10. 文档编写原则（强制）

> ⚠️ **强制要求**
>
> 所有技术规范文档必须遵守以下原则，违反视为不合格：

### 10.1 章节编号格式（强制）

**所有文档的章节标题必须使用阿拉伯数字编号**，禁止使用中文数字：

| 错误 | 正确 |
|:-----|:-----|
| `## 一、xxx` | `## 1. xxx` |
| `## 十、xxx` | `## 10. xxx` |
| `## 第1章` | `## 1. xxx` |

**格式要求**：
- 一级标题：`## 1.`
- 二级标题：`### 1.1`
- 阿拉伯数字 + 半角点号 + 空格

**示例**：
```markdown
## 1. 项目概述

## 2. 功能需求

### 2.1 用户管理
```

### 10.2 规范文档标题标注（强制）

**规范文档**（docs/技术规范/ 下的所有文档）的章节标题标注规则：

#### 通用规范文档

| 章节类型 | 示例 | 是否标注强制 |
|:---------|:-----|:-------------|
| 描述性章节 | `## 1. 概述`、`### 1.1 适用范围` | ❌ 不标注 |
| 规范性质章节 | `## 2. 命名规范（强制）`、`### 3.1 禁止行为（强制）` | ✅ 标注 |

> **判断标准**：章节内容是否包含"必须"、"禁止"、"强制要求"等规范性质的要求

适用文档：`docs/技术规范/` 下除技术锁定类规范外的所有文档。

#### 技术锁定类规范

以特定技术命名的规范文档（如 Vue前端规范.md、Flask后端规范.md、爬虫规范.md 等），因其内容本身就是技术要求，**所有章节均可标注强制**，无需逐章判断。

**非规范文档**（需求文档、设计文档等）**不需要标注强制**。

### 10.3 章节组织原则（强制）

**相同类型的规范必须放在同一章节，禁止分散**：

| 错误做法 | 正确做法 |
|:---------|:---------|
| 路径规范放在第3章，下拉响应格式放在第9章 | 路径规范和下拉响应格式都放在第3章 |
| 同一类接口的规范分散在多个章节 | 同一类接口的规范集中在一个章节 |
| 相关字段说明分散在不同章节 | 相关字段说明集中在同一章节 |

**章节编号必须连续，不得跳跃**：
- ❌ 3.2 → 3.4（缺少3.3）
- ✅ 3.1 → 3.2 → 3.3

### 10.4 不冗余原则（强制）

**禁止在多个文档中重复定义相同内容**。每个知识点只在一个地方定义，其他地方通过引用链接。

| 错误做法 | 正确做法 |
|:---------|:---------|
| 在多个文档中写相同的概念定义 | 概念只定义一次，其他地方引用 |
| 在 AI操作规范 中写详细的技术细节 | 详细内容写在技术规范中，AI操作规范只引用 |
| 在多个地方复制粘贴相同的说明 | 写一份，其他地方引用 |

### 10.5 不重复定义原则（强制）

**同一个概念、规则、流程只定义一次**，无论是在单个文档内部还是多个文档之间。

**单个文档内部**：
- 同一概念只在文档的一处定义
- 相似内容合并为一处，避免分散描述
- 避免"在第3章讲过，第5章又讲一遍"

**多个文档之间**：
- 详细内容只在专属文档中定义，其他文档只引用
- 避免在多个文档中复制相同的说明

### 10.6 内容完整原则（强制）

**每个文档的内容必须是完整的**，避免：

- 文档内容残缺，需要参照其他文档才能理解
- 关键信息缺失，需要猜测或询问
- 示例不完整，无法直接使用

### 10.7 适用范围标注规范（强制）

**同一文档包含多种技术时，必须在章节标题标注适用范围**。

| 标注方式 | 示例 |
|:---------|:-----|
| 技术无关 | `## 2. ORM框架（MySQL/SQLAlchemy）` |
| 纯通用 | `## 5. 反向代理（通用）` |
| 多技术并列 | `## 3. 缓存（Redis/Memcached）` |

**规则**：
- 章节内容仅涉及单一技术 → 在标题标注该技术
- 章节内容通用 → 在标题标注"（通用）"
- 章节内容涉及多种技术 → 在标题标注所有适用技术

**示例**：
```markdown
## 2. 数据库规范

### 2.1 ORM框架（MySQL/SQLAlchemy）
### 2.2 MongoDB（规划中）

## 3. 缓存规范

### 3.1 Redis（通用）
### 3.2 Memcached（规划中）

## 4. 部署规范

### 4.1 Docker（通用）
### 4.2 Kubernetes（规划中）
### 4.3 Python镜像（Python）
```

**好处**：
- 用户一看标题就知道内容是否适用
- 避免通用规范被误认为适用于所有技术
- 新增技术时只需在标题添加，无需重构文档

### 10.8 编写检查清单（强制）

编写新文档或更新文档时，对照检查：

- [ ] 是否有内容在其他文档中重复定义？
- [ ] 是否有内容应该引用而非复制？
- [ ] 文档内容是否完整，可以独立使用？
- [ ] 示例是否完整可运行？
- [ ] 引用路径是否正确？
- [ ] 同一文档内多技术是否已标注适用范围？
