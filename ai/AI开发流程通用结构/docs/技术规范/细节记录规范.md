# 细节记录规范

本文档定义 `docs/细节记录/` 目录的使用规范。

---

## 1. 概述

### 1.1 定位

`docs/细节记录/` 是项目内共享的记忆文件，用于记录用户告知的重要信息，确保团队所有成员和 AI 都能知晓并遵守。

### 1.2 与 Claude Code memory 的区别

| 机制 | 存储位置 | 共享方式 | 内容类型 |
|:-----|:---------|:---------|:---------|
| 细节记录 | `docs/细节记录/` | ✅ Git 共享 | 团队需知的信息 |
| Claude Code memory | `C:\Users\PC\.claude\` | ❌ 本地独享 | 私人偏好、AI 学习信息 |

---

## 2. 触发条件（强制）

**必须立即记录的情况**：

| 场景 | 示例 |
|:-----|:-----|
| 用户明确告知的限制 | "这个接口只能每天调用100次" |
| 用户明确的要求 | "密码必须使用AES加密" |
| 特殊业务规则 | "订单创建后不允许修改金额" |
| 环境配置信息 | "测试环境数据库地址是..." |
| 第三方集成要求 | "对接XX系统需要使用特定的签名算法" |
| 项目约定 | "所有日期字段使用时间戳" |

**不需要记录的情况**：
- 常规对话、问答
- 已在规范文档中定义的内容
- 明显的错误纠正

---

## 3. 记录格式（强制）

### 3.1 文件命名

```
docs/细节记录/{类别}-{日期}.md
```

**命名示例**：
```
docs/细节记录/需求限制-20240101.md
docs/细节记录/安全要求-20240115.md
docs/细节记录/接口约束-20240201.md
```

### 3.2 文件模板

```markdown
# {类别名称}

## 记录的信息

| 日期 | 内容 | 用途/关联 | 记录人 |
|-----|------|----------|--------|
| YYYY-MM-DD | {具体内容描述} | {用于什么功能/模块} | AI |

## 详情

### {日期}
{详细说明，包括：}
- 具体要求
- 原因背景
- 适用范围
```

### 3.3 示例

```markdown
# 接口约束

## 记录的信息

| 日期 | 内容 | 用途/关联 | 记录人 |
|-----|------|----------|--------|
| 2024-01-15 | 用户模块接口每日调用上限100次 | UserService | AI |

## 详情

### 2024-01-15
用户告知：用户模块的第三方接口有调用限制。

**具体要求**：
- 每个接口每日调用不超过 100 次
- 超出限制返回错误码 `10002`

**原因**：第三方接口有频控限制

**适用范围**：UserService 下所有调用该第三方接口的方法
```

---

## 4. 维护要求（强制）

### 4.1 记录时机

- **立即记录**：用户告知重要信息时，必须立即记录
- **不得拖延**：不允许"等会儿再记"，防止遗忘

### 4.2 更新机制

| 操作 | 要求 |
|:-----|:-----|
| 新增 | 创建新文件或追加到已有类别 |
| 修改 | 在详情中补充说明，保留历史记录 |
| 删除 | 不允许删除，历史记录应保留 |

### 4.3 格式检查

记录时应检查：
- [ ] 日期格式正确（YYYY-MM-DD）
- [ ] 内容描述清晰
- [ ] 用途/关联已标注
- [ ] 详情中有完整说明

---

## 5. AI 执行要求（强制）

### 5.1 开发前检查

每次修改代码前，必须检查 `docs/细节记录/` 目录：

```bash
ls docs/细节记录/*.md
```

若存在相关记录，必须遵守记录内容。

### 5.2 记录触发

用户告知信息时，执行以下判断：

```
用户告知 → 判断是否涉及限制/要求/约定 → 是 → 立即记录
                                            → 否 → 继续对话
```

### 5.3 自检确认

- [ ] 是否检查了 `docs/细节记录/` 目录？
- [ ] 是否有相关的记录需要遵守？
- [ ] 是否需要新增记录？

---

## 6. 与其他文档的关系（强制）

### 6.1 禁止重复

细节记录中的内容**不应**出现在其他规范文档中。若发现重复，应：
- 评估哪份文档更权威
- 删除重复内容，保留一处
- 若两份都需保留，说明原因

### 6.2 优先级

当细节记录与规范文档冲突时：
1. 用户明确说明的 → 以细节记录为准
2. 规范文档定义的标准 → 以规范文档为准
