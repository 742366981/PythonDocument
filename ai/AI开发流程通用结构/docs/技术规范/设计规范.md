# 设计规范

本文档定义需求文档和设计文档的编写标准，确保需求到设计的转换过程规范、可追踪。

---

## 1. 需求文档规范

### 1.1 需求文档必须包含的章节（基础框架）

| 章节 | 说明 | 强制 |
|:-----|:-----|:-----|
| 项目概述 | 名称、背景、目标 | 是 |
| 系统定位 | 边界、核心功能、用户角色 | 是 |
| 功能需求 | 模块划分、功能点列表 | 是 |
| 非功能需求 | 性能、安全等（如有） | 否 |
| 数据字典 | 核心数据实体及关系 | 是 |

> ⚠️ **章节扩展原则**：以上章节为需求文档的基础框架。当新增技术规范时（如消息队列规范、缓存规范），应在对应章节下增加相关内容，无需新增独立章节。
>
> **扩展示例**：
> - 若项目涉及缓存 → 在"非功能需求"下增加缓存需求说明
> - 若项目涉及消息队列 → 在"系统定位"或新增专项章节

### 1.2 需求提取原则（强制）

**原始需求可能以多种形式出现**（文字描述、截图、手写、语音转录等），必须确保转换后的需求文档精确完整：

| 提取原则 | 说明 |
|:---------|:-----|
| **不遗漏** | 原始需求中的每个功能点、每个字段都必须体现在需求文档中 |
| **不添加** | 需求文档只描述原始需求中的内容，不自行添加未提及的功能 |
| **可追溯** | 每个需求点应能追溯到原始需求的来源（文件/页面位置/时间） |

**提取检查清单**（每个功能点必须确认）：

```
□ 该功能在原始需求中有明确描述吗？
□ 该功能的输入/输出参数在原始需求中有定义吗？
□ 该功能的业务规则在原始需求中有说明吗？
□ 该功能涉及哪些字段？每个字段的处理方式明确吗？
□ 该功能是否有异常/边界情况？在原始需求中有描述吗？
```

### 1.3 需求提取示例

**原始需求示例**（用户提供的各种形式）：

```
用户说：订单管理要显示订单状态，是下拉选择的，有待支付、已支付、已发货、已完成、已取消这几种。
另外客户说订单备注以后可能要给用户看，所以可能也要做成可配置的。
```

**转换为需求文档**：

```markdown
### 订单管理

#### 订单列表
- 接口：GET /{项目前缀}/order/list
- 功能说明：显示订单列表，支持按状态筛选

| 参数名 | 类型 | 必填 | 说明 | 来源 |
|:-------|:-----|:-----|:-----|:-----|
| status | int | 否 | 订单状态下拉 | 基础数据表 order_status |
| page | int | 否 | 页码 | 写死（默认1） |
| page_size | int | 否 | 每页条数 | 写死（默认20） |

| 响应字段 | 类型 | 说明 | 来源 |
|:---------|:-----|:-----|:-----|
| id | int | 订单ID | 写死 |
| status | int | 订单状态 | 基础数据表 order_status |
| remark | str | 订单备注 | ⚠️ 待确认：是否需要基础数据表 |

#### 订单状态字段
- 字段名：status
- 类型：下拉框
- 枚举值：待支付、已支付、已发货、已完成、已取消
- 处理方式：✅ 基础数据表（可扩展）
- 获取接口：GET /{项目前缀}/base_data/order_status/

#### 订单备注字段（⚠️ 待确认）
- 用户原话："订单备注以后可能要给用户看，所以可能也要做成可配置的"
- 当前建议：做成基础数据表
- 状态：待二次确认
```

### 1.4 字段识别要求（强制）

**在需求分析阶段，必须识别以下类型的字段，并标注处理方式**：

| 字段类型 | 识别特征 | 处理方式 |
|:---------|:---------|:---------|
| **下拉框字段** | 枚举值明确但可能扩展、用户从列表选择 | 基础数据表 |
| **可配置参数** | 明确说了"后台配置"/"可配置"/"可维护" | 基础数据表 |
| **业务字典** | 状态、类型、分类、标签等 | 基础数据表 |
| **写死的值** | 当前写死但未来可能变化 | 基础数据表（预判） |
| **纯静态值** | 明确不会变化、不会扩展 | 写死 |

**识别示例**：

```markdown
### 订单状态
- 字段名：status
- 类型：下拉框
- 当前枚举值：待支付、已支付、已发货、已完成、已取消
- 处理方式：✅ 基础数据表（状态可能新增/修改）
- 接口：GET /{项目前缀}/base_data/order_status/
```

## 2. 设计文档规范

### 2.1 设计文档必须包含的章节（基础框架）

| 章节 | 说明 | 强制 |
|:-----|:-----|:-----|
| 系统概述 | 项目定位、主要功能 | 是 |
| 技术选型 | 后端/前端/数据库/缓存等 | 是 |
| 技术规范 | 遵循的规范清单 | 是 |
| 系统架构 | 模块划分、目录结构 | 是 |
| 接口设计 | 核心接口列表 | 是 |
| 数据字典 | 表结构、字段说明 | 是 |
| 基础数据模块设计 | 基础数据表结构、接口设计 | 是 |

> ⚠️ **章节扩展原则**：以上章节为设计文档的基础框架。当新增技术规范时（如新增了"消息队列规范"），应在对应章节下增加相关内容：
> - 若涉及消息队列 → 在"系统架构"下增加消息队列架构设计
> - 若涉及缓存 → 在"系统架构"下增加缓存架构设计
> - 若涉及定时任务 → 在"接口设计"或新增专项章节
>
> **核心思想**：设计文档框架稳定，新增技术通过扩展现有章节实现，无需修改本规范。

### 2.2 项目类型识别（强制）

**在设计阶段，必须根据需求识别项目类型**，并匹配对应的技术规范：

| 项目类型 | 识别依据 | 必须遵守的技术规范 |
|:---------|:---------|:------------------|
| 后端 + Flask | 需求涉及 HTTP API、后端逻辑 | API规范.md、Flask后端规范.md |
| 前端 + Vue | 需求涉及页面、组件、路由 | Vue前端规范.md |
| 全栈 | 同时包含后端和前端 | 按实际技术栈确定 |
| 爬虫 | 需求涉及数据采集、爬取 | 爬虫规范.md |
| 脚本/工具 | 需求为离线工具、脚本 | 按实际技术确定 |

**识别流程**：

```
1. 分析需求文档中的功能描述
2. 判断涉及的技术领域（API/页面/爬取/脚本）
3. 确定项目类型组合
4. 对照技术规范清单
5. 在设计文档中列出必须遵守的规范
```

### 2.3 技术选型流程（强制）

**技术选型必须按以下流程确认**：

```
┌─────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────┤
│  1. 列出可选技术栈                                         │
│     - 后端框架选项（Flask/Node.js/Go/Java等）              │

│     - 数据库选项（MySQL/PostgreSQL/MongoDB等）             │
│     - 缓存选项（Redis/Memcached等）                        │
├─────────────────────────────────────────────────────────────┤
│  2. 询问用户确认                                           │
│     - 提供推荐选项及理由                                    │
│     - 说明各选项的适用场景                                  │
├─────────────────────────────────────────────────────────────┤
│  3. 确认遵循的规范                                         │
│     - 检查 docs/技术规范/ 是否有对应规范                    │

```

### 2.4 模块划分原则

**模块划分应遵循以下原则**：

| 原则 | 说明 |
|:-----|:-----|
| 单一职责 | 每个模块只负责一类业务 |
| 高内聚低耦合 | 模块内部紧密相关，模块间依赖最小化 |
| 层次清晰 | 按数据层→业务层→接口层分层 |
| 基础数据独立 | 基础数据模块单独划分，不混入业务模块 |

### 2.5 基础数据模块设计（强制）

**基础数据模块必须单独设计**，包含以下内容：

#### 2.5.1 基础数据表结构

| 表名 | 说明 | 字段说明 |
|:-----|:-----|:---------|
| order_status | 订单状态 | id, code, name, sort_order, is_active |
| product_category | 商品分类 | id, code, name, parent_id, level, sort_order, is_active |

**表结构设计要求**：
- 必须包含 `id`（主键）、`code`（编码）、`name`（名称）
- 必须包含 `sort_order`（排序）、`is_active`（是否启用）用于管理
- 级联数据（如分类）必须包含 `parent_id`、`level`

> ⚠️ **表结构详细设计请参考** `数据库规范.md` 第3章

#### 2.5.2 基础数据接口设计

| 接口模式 | 方法 | 路径 | 说明 |
|:---------|:-----|:-----|:-----|
| 列表 | GET | `/{项目前缀}/base_data/{表名}/` | 获取所有启用的基础数据 |
| 详情 | GET | `/{项目前缀}/base_data/{表名}/{id}/` | 获取单条基础数据 |
| 创建 | POST | `/{项目前缀}/base_data/{表名}/` | 新增基础数据（管理后台用） |
| 更新 | PUT | `/{项目前缀}/base_data/{表名}/{id}/` | 更新基础数据（管理后台用） |
| 删除 | DELETE | `/{项目前缀}/base_data/{表名}/{id}/` | 删除基础数据（管理后台用） |

> ⚠️ **接口详细设计请参考** `API规范.md` 第3章（通用接口规范）

**列表接口响应格式**：

```json
{
  "code": 0,
  "message": "success",
  "data": [
    {
      "id": 1,
      "code": "pending",
      "name": "待支付"
    },
    {
      "id": 2,
      "code": "paid",
      "name": "已支付"
    }
  ]
}
```

### 4.3 通用规范关系

| 相关规范 | 关系说明 |

| `API规范.md` | 设计文档中的接口设计必须符合API规范 |

| `代码同步修改规范.md` | 设计文档变更必须同步更新需求文档和代码 |

### 4.4 新增规范时的扩展指南

**当新增技术规范时（如新增"消息队列规范"），设计规范本身无需修改**，只需在具体项目的设计文档中扩展：

| 新增规范 | 在设计文档中的扩展位置 | 扩展方式 |
|:---------|:---------------------|:---------|
| 消息队列规范 | "系统架构"章节 | 增加消息队列架构说明 |
| 缓存规范 | "系统架构"章节 | 增加缓存架构说明 |
| 定时任务规范 | "接口设计"或新增"定时任务设计"章节 | 增加定时任务列表 |
| 日志规范 | "系统架构"章节 | 增加日志收集方案 |
| 安全规范 | "非功能需求"章节 | 增加安全设计说明 |

**扩展示例**（假设新增了消息队列规范）：

```markdown
## 3. 系统架构

### 3.1 模块划分
{原有内容}

### 3.2 消息队列架构（⚠️ 新增）
> ⚠️ **详细设计请参考** `消息队列规范.md`

| 队列名称 | 类型 | 用途 | 生产者 | 消费者 |
|:---------|:-----|:-----|:-------|:-------|
| order_created | RabbitMQ | 订单创建事件 | 订单服务 | 库存服务、通知服务 |

### 3.3 目录结构
{原有内容}
```

**原则**：设计文档是"框架+扩展"的模式，框架稳定，扩展随技术规范新增而增加。设计规范本身保持稳定，不需要因为新增技术规范而修改。
