# 代码规范

本文档定义高质量代码的编写规范，适用于所有编程语言和项目类型（后端、前端、移动端、嵌入式等）。

> **核心原则**：写出可维护、可扩展、可测试的高质量代码
>
> **语言无关**：规范内容适用于任何编程语言，代码示例仅作参考

---

## 1. 数据建模规范（强制）

### 1.1 实体关联原则

**优先建立实体间关联，避免数据冗余**：

| 关联类型 | 正确做法 | 错误做法 |
|:---------|:---------|:---------|
| 一对多 | 主表存外键，从表关联主表 | 主表冗余从表数据 |
| 多对多 | 使用中间表关联 | 冗余 ID 列表 |
| 一对一 | 从表存主表外键 | 数据完全重复 |

```
# 错误示例：数据冗余
用户表 users
├── id
├── username
├── role_id        # 外键
├── role_name      # ❌ 冗余！从 roles 表关联即可
└── role_desc      # ❌ 冗余！

# 正确示例：外键关联
用户表 users
├── id
├── username
└── role_id        # 外键 → roles.id

角色表 roles
├── id
├── name
└── description
```

### 1.2 冗余数据控制

**允许冗余的条件（必须同时满足）**：

1. 该字段被查询的频率**极高**
2. 关联实体的变更频率**极低**
3. 冗余字段占用空间**可接受**
4. 数据一致性由**业务层保证**

| 场景 | 冗余是否合理 | 原因 |
|:-----|:-------------|:-----|
| 订单冗余用户昵称 | ✅ 合理 | 订单必然展示，昵称变更少 |
| 订单冗余用户电话 | ⚠️ 可选 | 电话可能变更，需权衡 |
| 订单冗余用户地址 | ❌ 不合理 | 地址频繁变更，关联查询不贵 |

### 1.3 关联查询的正确方式

**预加载关联数据，避免重复查询**：

```
❌ 错误：N+1 查询
   查询 100 条订单 → 再查询 100 次用户 = 101 次数据库调用

✅ 正确：预加载
   查询 100 条订单及其用户 → 1 次数据库调用（含 JOIN）
```

## 2. 查询性能规范（强制）

### 2.1 N+1 查询问题（强制）

**N+1 查询**：1 次主查询 + N 次关联查询

| 问题程度 | 查询次数 | 影响 |
|:---------|:---------|:-----|
| 严重 | 1 + N | 100 条记录 = 101 次查询 |
| 轻微 | 1 + 1 | 2 次查询，但可能无索引 |

**典型错误模式**：

```python
# ❌ 错误：循环内查询（N+1）
for order in orders:
    user = db.query("SELECT * FROM users WHERE id = ?", order.user_id)
    print(order.no, user.name)

# ✅ 正确：JOIN 预加载
orders = db.query("""
    SELECT o.*, u.name as user_name
    FROM orders o
    LEFT JOIN users u ON u.id = o.user_id
    LIMIT 100
""")

# ✅ 正确：批量查询后映射
user_ids = [o.user_id for o in orders]
users = db.query("SELECT * FROM users WHERE id IN (?)", user_ids)
user_map = {u.id: u for u in users}
for order in orders:
    print(order.no, user_map[order.user_id].name)
```

```javascript
// JavaScript/Node.js 同样存在 N+1

// ❌ 错误：循环内查询
for (const order of orders) {
    const user = await db.query('SELECT * FROM users WHERE id = ?', order.userId);
    console.log(order.no, user.name);
}

// ✅ 正确：预加载
const orders = await db.query(`
    SELECT o.*, u.name as userName
    FROM orders o
    LEFT JOIN users u ON u.id = o.userId
`);
```

### 2.2 深度分页性能问题

```sql
-- ❌ 错误：OFFSET 越大越慢
SELECT * FROM orders ORDER BY id LIMIT 20 OFFSET 1000000;

-- ✅ 正确：游标分页
SELECT * FROM orders
WHERE id > 1000000
ORDER BY id LIMIT 20;

-- ✅ 正确：限制最大 OFFSET
SELECT * FROM orders ORDER BY id LIMIT 20 OFFSET 1000;  -- 最大翻到 50 页
```

### 2.3 批量操作规范

**批量操作优先于循环单条操作**：

```python
# ❌ 错误：循环单条操作
for item in items:
    db.insert('orders', item)
db.commit()

# ✅ 正确：批量插入
db.batch_insert('orders', items)
db.commit()

# ❌ 错误：循环单条更新
for item in updates:
    db.update('orders', {'status': item.status}, {'id': item.id})

# ✅ 正确：批量更新
db.query("""
    UPDATE orders
    SET status = CASE id
        WHEN 1 THEN 'pending'
        WHEN 2 THEN 'paid'
        ...
    END
    WHERE id IN (1, 2, ...)
""")
```

### 2.4 查询字段控制

**禁止查询不需要的字段**：

| 错误做法 | 正确做法 |
|:---------|:---------|
| `SELECT *` | 只查询需要的字段 |
| 返回完整对象 | 按需返回部分字段 |
| 大字段无条件加载 | 大字段延迟加载或单独查询 |

## 3. 函数设计规范（强制）

### 3.1 单一职责原则（SRP）

**一个函数只做一件事**：

```python
# ❌ 错误：职责过多
def create_order(data):
    validate(data)        # 验证
    calculate(data)       # 计算
    save(data)            # 保存
    notify(data)          # 通知
    log(data)             # 日志
    return data

# ✅ 正确：职责分离
def validate_order_data(data):
    """验证订单数据"""
    pass

def calculate_order(data):
    """计算订单"""
    pass

def create_order(data):
    """创建订单（核心逻辑）"""
    validate_order_data(data)
    data = calculate_order(data)
    save(data)
    return data

def create_order_with_notification(data):
    """创建订单并发送通知"""
    order = create_order(data)
    notify_async(order.id)
    return order
```

### 3.2 参数规范

| 要求 | 说明 |
|:-----|:-----|
| 参数数量 ≤ 3 | 超过3个应使用配置对象 |
| 参数类型明确 | 使用类型声明 |
| 避免可变默认参数 | 使用 `None` 而非 `[]` 或 `{}` |

```python
# ❌ 错误：参数过多
def create_user(name, age, email, phone, address, role_id, status, source, tags, remark):
    pass

# ✅ 正确：使用配置对象
class CreateUserInput:
    name: str
    email: str
    phone: str
    address: str
    role_id: int
    status: int = 1
    source: str = 'web'
    remark: str | None = None

def create_user(input: CreateUserInput):
    pass
```

```javascript
// JavaScript 使用对象解构
// ❌ 错误
function createUser(name, age, email, phone, address, roleId, status, source, tags) {}

// ✅ 正确
function createUser({ name, age, email, phone, address, roleId, status = 1, source = 'web' }) {}
```

### 3.3 返回值规范

| 要求 | 说明 |
|:-----|:-----|
| 返回类型一致 | 成功/失败返回同类型或使用 Result 封装 |
| 不返回 null 表示成功 | 使用特定值或异常 |
| 集合类型返回空集合 | 返回 `[]` 而非 `null` |

```typescript
// ❌ 错误：返回类型不一致
function getUser(id: number): User | null | boolean {
    if (!id) return null;
    const user = db.find(id);
    if (!user) return false;  // 类型不一致
    return user;
}

// ✅ 正确：使用 Result 封装
interface Result<T> {
    success: boolean;
    data?: T;
    error?: string;
}

function getUser(id: number): Result<User> {
    if (!id) return { success: false, error: 'Invalid ID' };
    const user = db.find(id);
    if (!user) return { success: false, error: 'User not found' };
    return { success: true, data: user };
}
```

## 4. 错误处理规范（强制）

### 4.1 异常使用原则

| 使用异常 | 不使用异常 |
|:---------|:-----------|
| 预期外的错误（网络中断、内存不足） | 业务校验失败（用返回值） |
| 程序无法继续执行 | 表单验证失败 |
| 需要中断调用栈 | 业务流程控制 |

```python
# ❌ 错误：用异常处理业务逻辑
def withdraw(amount):
    if amount > balance:
        raise ValueError('余额不足')  # 业务校验不应抛异常
    balance -= amount

# ✅ 正确：业务校验用返回值
def withdraw(amount):
    if amount > balance:
        return False, '余额不足'
    balance -= amount
    return True, None
```

### 4.2 异常捕获规范

**捕获具体异常，不过度捕获**：

```python
# ❌ 错误：过度捕获所有异常
try:
    result = do_something()
except Exception as e:
    log.error(e)  # 丢失了具体类型信息

# ✅ 正确：捕获具体异常
try:
    result = db.execute(stmt)
except ConnectionError as e:
    log.error(f'数据库连接失败: {e}')
    raise ServiceUnavailable('服务暂时不可用')
except IntegrityError as e:
    log.warning(f'数据完整性错误: {e}')
    raise DuplicateKeyError('数据已存在')
```

### 4.3 异常传播原则

**只传播必要信息，隐藏实现细节**：

```python
# ❌ 错误：暴露内部实现
try:
    db.query("SELECT ...")
except OperationalError as e:
    # ❌ 暴露了数据库类型 MySQL
    raise Exception(f'MySQL错误: {e.code}, {e.msg}')

# ✅ 正确：转换为通用异常
try:
    db.query("SELECT ...")
except Exception as e:
    log.error(f'查询失败: {e}')
    raise QueryError('查询数据失败')
```

## 5. 防御性编程规范（强制）

### 5.1 参数校验

**入口函数必须校验输入**：

```python
# ❌ 错误：信任外部输入
def delete_user(user_id):
    db.delete('users', {'id': user_id})

# ✅ 正确：校验参数
def delete_user(user_id: int) -> Result:
    if not user_id or not isinstance(user_id, int):
        return Result(success=False, message='Invalid ID')

    if not db.exists('users', user_id):
        return Result(success=False, message='User not found')

    if db.exists('orders', {'user_id': user_id}):
        return Result(success=False, message='Cannot delete user with orders')

    db.delete('users', {'id': user_id})
    return Result(success=True)
```

```javascript
// JavaScript 同样需要校验
// ❌ 错误
function deleteUser(userId) {
    db.delete('users', { id: userId });
}

// ✅ 正确
function deleteUser(userId) {
    if (!userId || typeof userId !== 'number') {
        return { success: false, message: 'Invalid ID' };
    }
    if (!db.exists('users', userId)) {
        return { success: false, message: 'User not found' };
    }
    db.delete('users', { id: userId });
    return { success: true };
}
```

### 5.2 空值处理

**链式调用前检查空值**：

```python
# ❌ 错误：链式调用不检查空值
city = get_user(user_id).profile.address.city

# ✅ 正确：安全获取
user = get_user(user_id)
if not user or not user.profile:
    return None
return user.profile.address.city

# ✅ 更安全：使用 Optional/Maybe 模式
def get_city(user_id):
    user = get_user(user_id)
    return user?.profile?.address?.city  # 类似 Swift/ Kotlin 的 optional chaining
```

### 5.3 并发问题

**更新时防止竞态条件**：

```python
# ❌ 错误：并发更新丢失
def update_stock(product_id, quantity):
    product = db.get('products', product_id)
    product.stock -= quantity  # 读取后修改，无锁
    db.update('products', product)

# ✅ 正确：乐观锁
def update_stock(product_id, quantity):
    product = db.get('products', product_id)
    result = db.execute("""
        UPDATE products
        SET stock = stock - ?, version = version + 1
        WHERE id = ? AND version = ?
    """, [quantity, product_id, product.version])
    if result.rows_affected == 0:
        raise ConflictError('Data has been modified, please retry')

# ✅ 正确：悲观锁
def update_stock(product_id, quantity):
    db.begin_lock()  # 开启事务锁
    try:
        product = db.get('SELECT * FROM products WHERE id = ? FOR UPDATE', product_id)
        if product.stock < quantity:
            raise ValueError('Insufficient stock')
        product.stock -= quantity
        db.update('products', product)
        db.commit()
    except:
        db.rollback()
        raise
```

## 6. 代码复用规范（强制）

### 6.1 DRY 原则（Don't Repeat Yourself）

**重复代码必须抽象**：

```python
# ❌ 错误：重复代码
def create_user_order(user_id, items):
    if not user_id: raise ValueError('User ID required')
    if not items: raise ValueError('Items required')
    for item in items:
        if not item['product_id']: raise ValueError('Product ID required')
        if item['quantity'] <= 0: raise ValueError('Quantity must be positive')
    # 创建订单...

def create_guest_order(guest_name, items):
    if not guest_name: raise ValueError('Guest name required')
    if not items: raise ValueError('Items required')
    for item in items:
        if not item['product_id']: raise ValueError('Product ID required')
        if item['quantity'] <= 0: raise ValueError('Quantity must be positive')
    # 创建订单...

# ✅ 正确：抽象公共验证
def validate_order_items(items):
    if not items:
        raise ValueError('Items required')
    for item in items:
        if not item.get('product_id'):
            raise ValueError('Product ID required')
        if not item.get('quantity') or item['quantity'] <= 0:
            raise ValueError('Quantity must be positive')

def create_user_order(user_id, items):
    if not user_id:
        raise ValueError('User ID required')
    validate_order_items(items)
    # 创建订单...

def create_guest_order(guest_name, items):
    if not guest_name:
        raise ValueError('Guest name required')
    validate_order_items(items)
    # 创建订单...
```

### 6.2 避免硬编码

```python
# ❌ 错误：魔法值
if status == 1:
    send_notification()
elif status == 2:
    complete_order()
elif status == 3:
    cancel_order()

# ✅ 正确：使用常量/枚举
class OrderStatus:
    PENDING = 1
    PAID = 2
    COMPLETED = 3
    CANCELLED = 4

if status == OrderStatus.PENDING:
    send_notification()
```

## 7. 注释与文档规范（强制）

### 7.1 必要注释的场景

| 场景 | 要求 |
|:-----|:-----|
| 复杂业务逻辑 | 必须注释解释 WHY |
| 非直观写法 | 必须注释解释原因 |
| 性能关键代码 | 必须注释说明性能考虑 |
| 外部依赖 | 必须注释说明来源和用途 |
| 未完成代码 | 使用 `TODO:` 标记 |

### 7.2 注释要求

```python
# ❌ 错误：无意义注释
def calculate_price(quantity, unit_price):
    # 计算价格
    return quantity * unit_price

# ✅ 正确：解释"为什么"而非"是什么"
def calculate_price(quantity, unit_price):
    """
    计算订单价格

    注意：折扣已在调用方预处理，此处不处理
    """
    return quantity * unit_price

# ✅ 正确：复杂逻辑分步注释
def optimize_query(base_query):
    # 第一步：应用用户可见的过滤条件
    query = query.filter(status == STATUS_ACTIVE)

    # 第二步：排除测试数据（测试数据由测试账号创建）
    query = query.exclude_test_data()

    # 第三步：应用隐私过滤（用户设置不公开的内容）
    query = query.apply_privacy_filter(current_user_id)

    return query
```

### 7.3 TODO 注释规范

```python
# ✅ 正确：TODO 格式
# TODO(username): 优化此处查询性能，计划在 v2.1 使用缓存
# TODO(username): 移除此临时方案，上游接口已修复
```

### 8.1 日志级别使用

|:-----|:---------|
| DEBUG | 开发调试，生产环境不开启 |

logger.error('Query failed')

logger.error(f'Query failed: user_id={user_id}, error={e}')

## 9. 安全规范（强制）

### 9.1 SQL 注入防护

**禁止字符串拼接 SQL**：

```python
# ❌ 错误：SQL 注入风险
query = f"SELECT * FROM users WHERE name = '{name}'"
db.execute(query)

# ✅ 正确：参数化查询
db.execute("SELECT * FROM users WHERE name = ?", [name])

# ✅ ORM 方式（推荐）
User.objects.filter(name=name).first()
```

```javascript
// JavaScript 同样需要防止注入
// ❌ 错误
const query = `SELECT * FROM users WHERE name = '${name}'`;

// ✅ 正确：参数化查询
const result = await db.query('SELECT * FROM users WHERE name = ?', [name]);
```

### 9.2 敏感数据处理

```python
# ❌ 错误：日志记录敏感信息
logger.info(f'Login: username={username}, password={password}')

# ✅ 正确：脱敏处理
logger.info(f'Login: username={username}')

# ✅ 正确：只记录非敏感字段
logger.info(f'Login: username={username}, attempt={attempt}')
```

## 10. 代码风格规范

### 10.1 命名规范

#### 10.1.1 基础类型命名

| 类型 | 规范 | 示例 |
|:-----|:-----|:-----|
| 变量 | 小写下划线 / 小驼峰 | `user_name`, `userName` |
| 函数/方法 | 小写下划线 / 小驼峰 | `get_user`, `getUser` |
| 类 | 大驼峰 | `UserService`, `OrderController` |
| 常量 | 全大写下划线 | `MAX_RETRY`, `API_KEY` |
| 私有成员 | 下划线前缀 | `_internal_method` |

#### 10.1.2 通用命名原则

命名应自解释，禁止无意义简写，项目内缩写需保持一致。

```python
# ✅ 正确示例
user_name, is_active, get_user_by_id

# ❌ 错误示例
func1        # 不可自解释
usr_nm       # 无意义简写
pd           # 歧义（product? paid? password?）
```

#### 10.1.3 布尔变量命名

| 前缀 | 含义 | 示例 |
|:-----|:-----|:-----|
| `is_` | 是否为某状态/条件 | `is_active`、`is_deleted`、`is_empty` |
| `has_` | 是否拥有某属性/关系 | `has_permission`、`has_child`、`has_error` |
| `can_` | 能否执行某操作 | `can_edit`、`can_delete`、`can_export` |
| `should_` | 是否应该做某事 | `should_retry`、`should_validate` |
| `need_` | 是否需要做某事 | `need_refresh`、`need_migration` |

### 10.2 代码格式化

| 要求 | 说明 |
|:-----|:-----|
| 缩进一致 | 4 空格或 2 空格，禁止混合 |
| 行长度 | ≤ 120 字符 |
| 空格使用 | 运算符前后、逗号后加空格 |
| 空行使用 | 函数间空一行，逻辑块间空一行 |

## 11. 检查清单

### 开发自检

| 检查项 | 要求 |
|:-------|:-----|
| 数据建模 | 是否正确使用外键？是否有不必要的冗余？ |
| N+1 查询 | 是否有循环内查询？是否使用了预加载？ |
| 分页 | 深度分页是否使用了游标分页？ |
| 参数校验 | 入口函数是否校验了所有输入？ |
| 错误处理 | 是否捕获了具体异常？是否隐藏了实现细节？ |
| 代码重复 | 相似逻辑是否已抽象为公共函数？ |
| 硬编码 | 魔法值是否已定义为常量？ |
| 日志 | 关键操作是否记录了日志？日志是否包含上下文？ |
| 安全 | 是否有注入风险？敏感数据是否泄露？ |

### Code Review 关注点
