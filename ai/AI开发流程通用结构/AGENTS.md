# AI 项目配置

本文档是 AI 项目的配置文件，定义 AI 在本项目中的行为约束和执行规范。Claude Code 工具自动加载此文件。

> **说明**：不同 AI 工具可能使用不同配置文件（`CLAUDE.md` 或 `AGENTS.md`），两者内容保持一致。
>
> **重要**：修改规范时，**必须同时更新两个文件**，确保内容一致。详见 `AI操作规范.md`。

---

## 会话首次行为

**当会话开始后收到第一条消息时，AI 必须先执行 Step 1 的完整流程，再响应用户需求**：

> ⚠️ **会话开始后的首条消息必须触发 Step 1，完成项目情况汇报后才能回答**
>
> ⚠️ **后续消息正常响应，不再重复执行 Step 1**
>
> ⚠️ **读取的所有规范必须严格遵守，不得违背其中任何规定**

**执行方式**：直接执行下方 **Step 1：识别核心规范** 的完整流程（见第3章）。

**验证标准**：
- [ ] 是否已扫描 `docs/技术规范/` 目录？ → 输出扫描结果
- [ ] 是否已读取所有适用的规范文件？ → 每个文件输出 `✓ 已读取：{文件路径}`
- [ ] 是否已向用户汇报完整项目情况？ → 必须包含所有规范清单
- [ ] 是否已执行环境检查？ → 输出实际命令执行结果
- [ ] 是否已总结核心要点？ → 必须包含3个强制要点

> ⚠️ **以上步骤必须完整执行，不得跳过任何步骤；读取规范后必须严格遵守**

## 0. 速查表（每次任务前必须对照）

### 强制执行项
- [ ] **修改前**：必须先分析影响、询问确认
- [ ] **修改时**：必须遵守 `docs/技术规范/代码同步修改规范.md`
- [ ] **完成时**：必须立即 commit（禁止多个操作后才 commit）
- [ ] **接口开发**：必须先写 docstring，再写实现
- [ ] **代码注释**：函数/类/复杂逻辑必须有完整注释
- [ ] **临时文件**：必须放在 `temp/` 目录，用完即删
- [ ] **细节记录**：用户告知关键信息时，必须立即记录到 `docs/细节记录/`

### 禁止项
- ❌ 未经用户确认直接修改任何代码/文档
- ❌ 先写代码后补文档
- ❌ 只 commit 代码不 commit 文档
- ❌ 多个文档重复定义相同内容
- ❌ 在项目目录创建临时文件

## 1. 强制行为约束（红线）

| 红线行为 | 违规后果 |
|:---------|:---------|
| 未经确认直接修改代码/文档 | 用户有权要求回滚 |
| 先写代码后补文档 | 视为不合格 |
| 多个操作后才 commit | 视为不规范 |
| 只 commit 代码不 commit 文档 | 视为不规范 |
| 发现重复定义未处理 | 视为不合格 |
| 代码注释不完整 | 视为不合格 |
| 临时文件不清理 | 视为不规范 |

## 2. 细节记录机制

### 触发条件
- 用户主动告知关键信息时 → 立即记录
- 用户告知的场景/限制/注意事项 → 记录并后续遵守

### 记录位置
`docs/细节记录/`

### 记录格式
```markdown
## 用户告知的关键信息

| 日期 | 内容 | 用途/关联 |
|-----|------|----------|
| 2026-05-09 | 用户告知的XXX | 用于XXX功能开发 |
```

### 开发前检查
- 修改某功能前 → 检查是否有该功能相关的用户告知记录
- 有记录 → 必须遵守记录内容
- 无记录 → 正常开发

## 3. 执行Checklist

### Step 1：识别核心规范（**强制执行**）

**触发条件**：**始终执行**

**执行内容**：

**1.1 扫描规范目录（强制执行）**

使用 Bash 工具**实际执行**以下命令：

```bash
ls docs/技术规范/*.md
```

**执行后必须输出扫描结果**，例如：
```
docs/技术规范/API规范.md
docs/技术规范/数据库规范.md
docs/技术规范/缓存规范.md
...
```

**1.2 确定项目类型和技术栈**

使用 Read 工具读取 `docs/设计文档/` 目录下的设计文档，提取完整技术栈信息：

**必须输出以下信息**：
- 项目类型：后端/前端/全栈/爬虫/其他
- 后端框架：如 Flask、FastAPI、Django 等
- 数据库：如 MySQL、PostgreSQL、MongoDB、Redis 等
- 前端框架（如有）：如 Vue、React、Element Plus 等
- 其他组件：如 Gunicorn、Nginx、Docker 等

**输出格式**：
```
项目类型：{类型}
技术栈：{框架} + {数据库} + {其他组件}
```

**1.3 识别适用的技术锁规范**

根据 Step 1.2 读取到的实际技术栈，按需匹配：

| 技术特征 | 技术锁规范 |
|:--------|:-----------|
| 后端框架含 Flask | `Flask后端规范.md` |
| 前端框架含 Vue | `Vue前端规范.md` |
| 涉及爬虫功能 | `爬虫规范.md` |
| 其他 | 按实际技术栈确定 |

**1.4 读取适用的规范文件（强制）**

根据项目类型，使用 Read 工具**依次读取**所有适用的规范文件：

| 项目类型 | 必须读取的规范文件 |
|:--------|:------------------|
| 所有项目 | `docs/技术规范/Git规范.md`、`docs/技术规范/代码同步修改规范.md`、`docs/技术规范/开发环境规范.md` |
| 后端 + Flask | 上述文件 + `docs/技术规范/API规范.md`、`docs/技术规范/Flask后端规范.md` |
| 前端 + Vue | 上述文件 + `docs/技术规范/Vue前端规范.md` |
| 爬虫 | 上述文件 + `docs/技术规范/爬虫规范.md` |

每读取一个文件，输出：`✓ 已读取：{文件路径}`

> ⚠️ **读取规范后必须严格遵守，不得违背其中任何规定**

**1.5 汇报项目情况（强制）**

向用户汇报以下内容：

```
## 项目情况汇报

### 1. 项目类型
- 类型：{后端/前端/全栈/其他}
- 技术栈：{识别的技术栈}

### 2. 识别到的所有规范
根据 Step 1.1 的扫描结果，**列出 `docs/技术规范/` 目录下的所有规范文件**：

| 序号 | 规范 | 文件 | 状态 |
|:----:|:-----|:-----|:-----|
| 1 | {从文件名提取} | {实际文件路径} | {当前项目适用/不适用} |
| 2 | {从文件名提取} | {实际文件路径} | {当前项目适用/不适用} |
| ... | ... | ... | ... |

> **说明**：列出所有规范，不管当前项目是否用到。"状态"列标注是否适用于本项目。

### 3. 本次需要遵守的规范
根据扫描结果确定：
- 通用规范：所有标注"当前项目适用"的规范
- 技术锁规范：{根据项目类型匹配的技术锁规范}

### 4. 细节记录检查
- 是否有相关记录：{是/否}
- 如有，需遵守的内容：{内容}
```

**自检确认**：
- [ ] 是否已扫描 `docs/技术规范/` 目录？ → 验证：能列出所有规范文件
- [ ] 项目类型是否已识别？ → 验证：明确项目类型
- [ ] 技术锁规范是否已定位？ → 验证：对应规范文件存在
- [ ] 是否已读取所有适用的规范？ → 验证：每个适用规范都有"✓ 已读取"输出
- [ ] 是否已向用户汇报？ → 验证：已发送项目情况汇报

⚠️ 若有任何一项为"否"，必须先完成该项才能继续。

**1.6 执行环境检查（强制）**

读取 `docs/技术规范/开发环境规范.md` 第 6.1 节后，**使用 Bash 工具实际执行相应的检查命令**（命令参考该文档），展示实际输出结果。

**执行后向用户确认**：
```
请确认环境检查结果是否正常：
- 环境版本：{实际输出}
- 依赖列表：{实际输出}
```

**发现问题必须立即向用户报告**，格式：
```
⚠️ 环境异常
- 问题：{描述}
- 处理方式：{提示用户操作}
```

**1.7 总结核心要点（强制）**

基于已读取的规范内容，总结本次项目必须遵守的核心要点，**必须包含以下内容**：

1. **细节记录机制**：用户告知关键信息时 → 立即记录到 `docs/细节记录/` 并后续严格遵守，不得违背
2. **代码同步修改规范**：变更必须同步所有相关文件
3. **Git commit 规范**：每个任务完成后立即 commit

完成以上步骤后，继续 Step 2。

### Step 2：查找需求
- [ ] `docs/原始需求/` 和 `docs/需求文档/` 是否已检查？

### Step 3：确认设计文档
- [ ] 设计文档是否存在？用户是否已确认技术选型？

### Step 4：代码执行
- [ ] 是否每个接口都先写了 docstring？
- [ ] 是否每个函数/类都有完整注释？
- [ ] 是否按要求 commit？

### Step 5：细节记录
- [ ] 本次开发是否涉及用户告知的关键信息？
- [ ] 如果是，是否已在 `docs/细节记录/` 中记录？

### Step 6：三阶段同步（强制）

需求、设计、代码三者必须保持同步，任何变更都必须向上下游传递。

**同步方向**：
| 同步方向 | 触发条件 |
|:---------|:---------|
| 正向同步 | 需求变更：原始需求 → 标准需求 → 设计文档 → 代码 |
| 逆向修正 | 代码实现与设计文档不符：代码 → 更新设计文档 |

**禁止行为**：
- ❌ 代码与设计文档不一致
- ❌ 设计文档与需求不一致
- ❌ 变更只改一处
- ❌ 跳过设计文档直接改代码

**验证检查清单**：
- [ ] 正向变更：需求→设计→代码 是否按顺序同步
- [ ] 逆向修正：代码与设计文档是否一致
- [ ] 跳级检查：是否有需求→代码跳过设计的情况

## 4. 项目结构

| 类型 | 路径 | 说明 |
|:-----|:-----|:-----|
| 虚拟环境 | `../venv/{项目名}-env/Scripts/python` | Windows：`Scripts/python`，Unix：`bin/python` |
| 原始需求 | `docs/原始需求/` | 用户提供的需求材料 |
| 需求文档 | `docs/需求文档/` | 标准化需求 |
| 设计文档 | `docs/设计文档/` | 架构和技术方案 |
| 技术规范 | `docs/技术规范/` | 技术标准规范 |
| API 文档 | `docs/API文档/` | 接口详细设计，**必须包含完整示例** |
| 细节记录 | `docs/细节记录/` | 用户告知的关键信息 |
| 临时文件 | `temp/` | 用完即删 |

### 4.1 目录职责

| 目录 | 职责 | 编写者 |
|:-----|:-----|:-------|
| 原始需求/ | 保存用户提供 | 用户 |
| 需求文档/ | 标准化需求 | AI |
| 设计文档/ | 定义架构和技术选型 | AI |
| 技术规范/ | 定义技术标准 | AI |
| API文档/ | 定义接口细节，**必须包含完整示例** | AI |
| 细节记录/ | 记录重要细节 | AI |

### 4.2 文档内容要求

**API文档/ 目录必须包含：**

| 文件 | 内容要求 | 参考示例 |
|:-----|:---------|:---------|
| swagger_template.md | Flasgger docstring 编写模板（**必须**） | `docs/API文档/swagger_template.md` |
| API文档.md | 完整接口文档（Markdown格式，**必须**） | `docs/API文档/API文档.md` |

> 注：swagger_spec.json 由 `tools/export_docs.py` 从代码 docstring 导出，无需预置

**内容深度：**
- 完整参数说明（含 type、description、example）
- 响应示例（成功/失败场景）
- 分页参数说明
- 导入导出文件格式说明
